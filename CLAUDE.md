## Overview

DayTrackr is an NX 19.6.x monorepo with two applications:
- **`apps/web`** — Angular 18.2.x SPA with NgRx state management
- **`apps/api`** — Spring Boot 3.0.6 REST API (Java 17) with MySQL and Flyway migrations

All Node/web dependencies live in the root `package.json`. API dependencies are managed by Maven via `apps/api/pom.xml`.

## Development Commands

### Java version

Before running API commands, activate Java 17 via SDKMAN:
```bash
sdk env
```

### Web app

```bash
npx nx serve web          # Dev server (localhost:4200)
npx nx build web          # Production build → dist/apps/web
npx nx test web           # Karma unit tests
npx nx lint web           # ESLint
```

### API

```bash
npx nx serve api          # Spring Boot dev server (localhost:8080, profile=dev)
npx nx build api          # Maven package (skip tests)
npx nx test api           # Maven test
npx nx lint api           # Lint (not configured — API has no linter target)
```

### Running a single API test class

```bash
cd apps/api && ./mvnw test -Dtest=HomeControllerTest
```

### Nx affected (used in CI)

```bash
npx nx affected --target=lint --parallel=3
npx nx affected --target=build --configuration=production
```

## Architecture

### Web (Angular + NgRx)

The store has four feature slices registered in `app.module.ts`:
- `auth` — JWT tokens, login/register state
- `days` — calendar day list and selected day
- `editDay` — in-progress edits to a day's events
- `profile` — user profile data
- `router` — synced via `@ngrx/router-store`

Each slice follows the pattern: `store/actions/*.actions.ts` → `store/effects/*.effects.ts` → `store/reducers/*.reducer.ts` → `store/selectors/*.selectors.ts`.

Route guards (`guards/auth.guard.ts`) protect all pages behind authentication. `guards/dirty-check.guard.ts` blocks navigation away from the profile page with unsaved changes. The `AuthInterceptor` (`interceptors/auth.interceptor.ts`) appends JWTs to every HTTP request.

Page components (e.g., `CalendarPageComponent`, `EditDayPageComponent`) act as NgRx-connected shells; presentational components beneath them receive data via `@Input`.

### API (Spring Boot)

Layered architecture: `controllers/` → `services/` → `repositories/` (Spring Data JPA).

Key layers:
- **Controllers** handle REST at `/api/*` — `AuthController`, `DailyEventController`, `FeedbackController`, `ProfileController`
- **Domain** — JPA entities (`UserEntity`, `DailyEvent`, `Event`, `FeedbackMessage`, `FileEntity`) and request/response models
- **Security** — stateless JWT filter chain in `security/SecurityConfig.java`; tokens generated by `JWTGenerator`; no OAuth2 resource server despite the pom dependency
- **Migrations** — Flyway manages all schema changes; add new scripts to `src/main/resources/db/migration/` following `V<n>__Description.sql` naming

### Spring Boot profiles

| Profile | Purpose | DB |
|---------|---------|-----|
| `dev` (default) | Local development | `localhost:3306/daytrackr`, username `aflorzy` |
| `prd` | Production | Configured via env vars |
| `ci` | CI test run | Defined in `application-ci.yml` |
| `test` | Integration test | Defined in `application-test.yml` |

The `dev` profile sets `ddl-auto: update` (overriding the base `validate`) and enables verbose SQL logging.

## Database Migrations

All schema changes go in `apps/api/src/main/resources/db/migration/` as `V<n>__Description.sql`. Flyway runs on startup. The dev profile has `clean-disabled: true` — do not run `flyway:clean` locally without explicitly uncommenting `clean-on-validation-error`.

## CI/CD

Gitea Actions (`.gitea/workflows/ci.yml`) runs two jobs:
- **PR validation** — lints and builds only affected projects
- **Main branch** — builds, pushes Docker images to the Gitea container registry, then triggers a staging deployment webhook

Docker images: `daytrackr-api` (eclipse-temurin:17-jdk-jammy + entrypoint.sh) and `daytrackr-web` (nginx:stable-alpine with SPA routing via `nginx.conf`).
